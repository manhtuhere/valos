From Prompt User to System Architect

Val OS — the agentic playground where you don't prompt, you direct.

A judgment-first operating system that turns raw ideas into typed architecture, routing decisions, and execution plans — with every decision visible, critiqued, and editable. Built so the next generation of system architects can see how a system thinks before they build one.

Track challenge → How might we build AI-native development tools or "agentic playgrounds" that move young women from prompt-engineering to building and directing system architecture?

01 · What it is

An agentic playground with a judgment loop on top.

The problem

Prompting is a dead-end career.

If the next wave of infrastructure is designed by one demographic, its biases get automated at scale. Architecture literacy — the ability to shape systems, not just query them — stays locked behind senior-engineer apprenticeship pipelines most young women never get access to.

The shift

Val OS makes architecture legible.

Every idea becomes an inspectable 11-stage pipeline: intent, reframe, scope, architecture, research, execution, routing, critic, output, memory. You watch the system reason, catch its own mistakes, and dispatch work to specialized agents. You learn architecture by directing one.

02 · Goal

Close the architecture gap — by exposure, not lectures.

Primary

Literacy through visibility

Turn the black box of "AI agents doing things" into a typed, readable pipeline. If you can see scope, routing, and critic decisions, you can critique them — and start making them yourself.

Secondary

Lower the floor, not the ceiling

A teenager with an idea should be able to produce the same structured plan a staff engineer would. The hard work of scoping and routing is done on the glass, in the open.

Tertiary

Build the habit of judgment

The critic gate teaches users that good outputs come from structured self-critique, not one-shot prompts. Revision loops are the product.

Loop A

Revision loop

If the critic returns revise, scope → architect → research → execution → router → critic re-runs with the critic's failures and fixes injected into context. Capped at 2×.

Loop B

Execution feedback

Routes fan out to Manus + OpenClaw in parallel via asyncio.gather. If any worker returns needs_revision, the critic re-runs with worker output attached.

Loop C

Memory writeback

Patterns, failures, and principles are proposed with a confidence score. Only entries above threshold (≥0.75) persist to the typed memory tables.

05 · ICP

Who this is for.

Primary

Young women, 17–28, moving up the stack

Students, bootcamp grads, and early-career builders who are fluent with AI tools but locked out of senior-architect mentorship. They want to design systems, not just prompt them — and they learn best by watching a system think out loud.

Secondary

Non-traditional builders

Self-taught devs, career switchers, and operators-turned-founders who skipped the traditional CS path. They need scaffolding for scoping, routing, and trade-off reasoning that a senior engineer would normally give over a whiteboard.

Tertiary

Solo founders

Any founder who needs a thinking partner that pushes back — not a yes-machine. The critic gate and revision loop produce the kind of structured self-challenge a cofounder would force on you.

06 · Features

What's in the box.

◆

Self-contained HTML playground

No install, no keys required. Demo mode runs the full 11-stage pipeline locally in the browser for instant architecture literacy.

⊞

Three modes, one file

Demo (local), Real API (browser-direct to Anthropic), or Backend (FastAPI orchestrator) — toggle in the header.

▦

Typed memory, scored retrieval

Six memory types: founder_principles, product_preferences, routing_rules, approved_patterns, failure_lessons, project_state — tag-scored with per-category caps.

⚖

Critic with shallow-prompt defense

Score-based gate (approve/revise/reject). Penalises vague prompts even when the bundle looks well-shaped — so "build a thing" can't coast on template quality.

↻

Auto revision loop

When the critic says revise, the pipeline re-runs with failures and fixes injected into context. Capped at 2× so it can't thrash.

⇆

Parallel worker fan-out

Research routes → Manus. Execution routes → OpenClaw. Dispatched concurrently via httpx + asyncio.gather. Pluggable: match the JSON contract, swap the URL.

◉

Execution feedback loop

Worker results flow back into the critic. Any needs_revision signal triggers a re-critique with worker logs attached.

▣

Postgres-backed memory

6 typed memory tables + audit table via docker-compose. Schema and PRD seeds auto-loaded on boot.

08 · README

How a judge (or contributor) runs it.

30-second demo (no install)

Open val-os.html in a browser. Demo mode is on by default — paste any prompt, watch 11 typed stages paint in real time. Toggle the auto-revise switch to watch the critic send the pipeline back for revision.

Full stack (four terminals)

# 1. Postgres + schema + seeds
cd val-os-backend
cp .env.example .env
docker compose up -d

# 2. Manus worker (research)
cd workers/manus && uvicorn main:app --port 8081

# 3. OpenClaw worker (execution)
cd workers/openclaw && uvicorn main:app --port 8082

# 4. Orchestrator
cd val-os-backend && uvicorn orchestrator.main:app --port 8080

Live smoke

curl -s http://localhost:8080/plan \
  -H 'Content-Type: application/json' \
  -d '{
    "raw_prompt": "Build me a founder operating system that turns prompts into product specs.",
    "auto_revise": true,
    "dispatch_workers": true
  }' | jq '{critic: .critic.status, score: .critic.score, revisions: .revisions}'

Offline smoke (no DB, no workers)

python3 scripts/smoke.py
# exits SMOKE OK · meta=approve@0.83 · "build a thing"=reject@0.58

What the judges should look at

  • Typed boundaries — orchestrator/schemas.py — every stage has a Pydantic model, which is why the pipeline is inspectable.
  • Critic penalty heuristic — orchestrator/brain.py → critic() — shallow-prompt defense that prevents a well-shaped bundle from masking a vague idea.
  • Revision loop — orchestrator/main.py → _run_pipeline() — the re-runs with critic fixes injected are the core learning loop.
  • Scored memory — orchestrator/memory.py → retrieve() — tag overlap × type weight + priority/confidence/severity bonuses, per-category caps.
  • Pluggable workers — workers/*/main.py — shape-correct stubs; swap to real Manus/OpenClaw by changing two env vars.
